# wiki-sync 产品规划

## 一句话定位

把 Claude Code、Codex 等 AI Agent 的对话，持续同步进 LLM-WIKI 的 `raw/`，喂给它的 ingest 工作流。

## 解决的问题

做并行项目时，人天天跟不同的 AI 聊产品思路、改代码、定方案。这些**零碎的知识和修改点**
散落在一个个聊天窗口里，关掉就找不回；不同 AI 对同一个项目的理解各不相同；PRD、设计文档
永远追不上真实的讨论。根因是：和 AI 的对话从没沉淀，更没汇入项目知识。

LLM-WIKI 本身有个工作流叫 **ingest**——把 `raw/` 里零散的素材自动编排、归纳进结构化的 wiki。
缺的只是一个稳定的"原料入口"：把分散在各个 AI 里的对话，源源不断地灌进 `raw/`。

**wiki-sync** 就是这个入口：

> 各 AI 对话里的零碎结论/思路/改动点 → 自动汇入 `raw/wiki-sync-<来源>/` → 经 ingest 编排进 wiki
> → 沉淀成可检索、可演进的项目知识。

它不是终点（存档），而是起点（喂给 ingest 的原料）。

## 目标用户

在 Obsidian 中使用 LLM-WIKI（含 ingest 工作流），并且日常用 Claude Code / Codex 等多个 AI 协作的人。

## 核心功能

### V0.1 — 最小可用 ✅ 已完成

| 功能 | 描述 | 状态 |
|------|------|------|
| 对话发现 | 直接扫描 Claude Code 的 jsonl 对话文件，列出可导入的对话 | ✅ |
| 单条导入 | 把指定对话转为 LLM-WIKI 格式的 Markdown，存入 vault | ✅ |
| 去重保护 | 已导入的对话不会重复导入 | ✅ |
| Log 更新 | 每次同步把运行结果（时间、新增/跳过/失败、导入清单）追加到 `raw/wiki-sync-log.md` | ✅ |

> 实现中调整：原计划读 `sessions-index.json` 索引，但实测它常和实际文件对不上，
> 改为直接扫 jsonl；原计划写 `wiki/index.md`，后改为按 agent 分文件夹（见下文）。

### V0.2 — 批量与自动化 ✅ 已完成

| 功能 | 描述 | 状态 |
|------|------|------|
| 批量导入 | `--all` 一键导入全部历史对话 | ✅ |
| 自动模式 | `wiki-sync install` 装进 agent，对话结束自动导入（push 模型） | ✅ |
| 自动摘要 | 从对话提取目标 + 结果 + 规模，写进 wiki 页面（提取式，不调 LLM） | ✅ |

### V0.3 — 多 Agent 支持 ✅ 已完成

| 功能 | 描述 | 状态 |
|------|------|------|
| Codex 适配 | 读取 `~/.codex/sessions/**/rollout-*.jsonl`，过滤系统注入 | ✅ |
| ChatGPT 适配 | 导入官网导出的 `conversations.json`（mapping 结构） | ✅ |
| 注册表架构 | 每个来源声明「路径 glob + 解析格式」，统一成对话模型 `conv` | ✅ |
| 按 agent 分文件夹 | 每个来源独立 `raw/wiki-sync-<source>/`，不污染主笔记 | ✅ |
| 自定义来源 | `source add` 接入任意新 agent（hermes/opencode 等），通用解析器兜底 | ✅ |

## 技术形态

### 核心：Python CLI（单文件、零依赖）

设计模型：**push —— 程序装进每个 agent，agent 聊完主动把对话推给它。** 换 agent 就再装一次。

```bash
wiki-sync install        # 装进当前 agent（自动识别 claude/codex）
wiki-sync install codex  # 或显式指定
wiki-sync status         # 看哪些 agent 装了
wiki-sync uninstall      # 卸载
```

辅助命令：

```bash
wiki-sync detect         # 列出本地 Obsidian 知识库（带笔记数），用户输序号选
wiki-sync                # 手动同步历史对话（首次 backfill）
wiki-sync list           # 看看有哪些对话
wiki-sync where <路径>    # 手动指定知识库位置
```

选择 Python 的理由：macOS 自带，零外部依赖即可运行，文本处理顺手。

### 各 agent 的接入方式

| Agent | 钩子机制 | 接入做法 |
|-------|---------|---------|
| Claude Code | Stop 钩子 | 往 `~/.claude/settings.json` 加 `wiki-sync hook-run`，事件含 transcript 路径从 stdin 传入 |
| Codex | `config.toml` 的 `notify` | 改成 `notify = ["wiki-sync","notify-run"]`，回合结束触发；取最新 rollout 同步 |

两处都**合并/接力写入，不破坏用户已有配置**：Claude 的其它 hook 保留；Codex 的 notify 只有一个槽位，
原有的（如 Computer Use）会被记下并在同步后接力调用，卸载时自动恢复。

> 早期版本用"中心化扫描 + 手动 `source add`"。后改为 push 模型——
> 不再让程序去翻各家存储目录，而是装进 agent、由 agent 主动推送，更干净也更实时。
> 扫描式 `source` 作为"手动导入历史"的兜底保留。

### 安装方式

```bash
# 用户只需一条命令
curl -sSL https://raw.githubusercontent.com/ruanjingwei/wiki-sync/main/install.sh | sh
```

卸载：

```bash
rm -f ~/.local/bin/wiki-sync
```

## 对话 → Wiki 的映射规则

各来源的对话存储位置：

```
Claude Code:  ~/.claude/projects/*/*.jsonl
Codex:        ~/.codex/sessions/**/rollout-*.jsonl
ChatGPT:      官网导出的 conversations.json（需手动导出）
```

> 注：Claude Code 直接扫 jsonl 文件，不用 `sessions-index.json`——
> 实测那个索引缓存常和实际文件对不上（记的 ID 没有对应文件）。

导入后，**每个 agent 在 `raw/` 下独立一个文件夹**，不碰 `wiki/` 里整理好的主笔记：

```
vault/
├── raw/
│   ├── wiki-sync-claude/
│   │   └── claude-<slug>.md     ← 一个对话一个文件（frontmatter + 摘要 + 全文）
│   ├── wiki-sync-codex/
│   ├── wiki-sync-<自定义>/       ← 新 agent 也各自一个文件夹
│   └── wiki-sync-log.md         ← 每次同步追加一条运行记录（时间 + 新增/跳过/失败 + 清单）
└── .wiki-sync/imported.json     ← wiki-sync 自己的去重记录（不碰库结构）
```

> 设计取舍（第一性原理）：早期版本会往 `wiki/sources/`、`wiki/index.md` 写，也会在每个
> `raw/wiki-sync-<source>/` 下维护一个 `_index.md`。后来索引文件全部去掉了——索引的本质是
> 检索手段，而 Obsidian 本身就负责索引和搜索，再手动维护一份 `_index.md` 只会过时、和实际
> 文件对不上、徒增噪声。所以 `raw/` 下**只放对话文件、不生成任何索引**，把自动同步的对话和
> 用户亲手整理的知识彻底分开，不污染主笔记。
>
> 同步日志也守同一条边界：放在 `raw/wiki-sync-log.md`，和导入的对话一样归在 `wiki-sync-*`
> 命名下，**绝不写进 `wiki/`** 等用户亲手整理的区域——wiki-sync 的产物全部自包含在 `raw/`
> （外加一份 `.wiki-sync/` 去重记录），不碰主笔记。日志本身守"操作必留痕"：一次同步就是一次
> 写库操作，每次都**无条件**追加一条运行记录（以"运行"为单位记时间、新增/跳过/失败数和导入
> 清单），空跑除外，便于日后审计和排错。

### 对话文件模板（摘要 + 全文合一）

```markdown
---
type: source
title: <对话标题>
tags: [<来源>-conversation, ai-chat]
related: []
created: YYYY-MM-DD
updated: YYYY-MM-DD
authors: [<来源显示名>]
venue: "<来源显示名>"
---

# <对话标题>

## 对话信息
- **来源**: <Claude Code / Codex / ChatGPT / 自定义>
- **会话 ID**: <id>
- **日期**: YYYY-MM-DD
- **消息数**: N
- **模型**: <model-name>

## 摘要
- **目标**: <首个用户问题>
- **结果**: <末条助手回复的前两句>
- **规模**: N 轮用户提问 · M 次工具调用

## 涉及话题
- ...

## 完整对话
### 🧑 用户
...
### 🤖 助手
...
```

> 早期版本把「结构化摘要」和「完整原文」拆成两个文件（wiki/ 一份、raw/ 一份）。
> 现在合并为一个文件，放在 `raw/wiki-sync-<source>/` 下，摘要在上、全文在下。

## 和同类产品的区别

| | wiki-sync | Mem0 | 手动复制粘贴 |
|---|---|---|---|
| 目标 | 对话 → Obsidian wiki | 通用记忆层 | — |
| 存储位置 | 你的本地 vault | 云端 | 你的 vault |
| 格式 | LLM-WIKI schema | 自有格式 | 随意 |
| 自动化 | 装进 agent，聊完自动同步（push） | API | 无 |
| 数据主权 | 完全本地 | 云端服务 | 完全本地 |

## 技术依赖

- **Python 3.9+**: macOS 自带，其他系统需安装
- **Obsidian 知识库**: 一个含 `raw/` 文件夹的 Obsidian 仓库（有 `.obsidian/`）即可，不要求 `.llm-wiki` 等特定结构
- **Claude Code / Codex** (可选): 使用自动同步时，在对应 agent 里 `install` 一次
